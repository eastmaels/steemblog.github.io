雪景

昨天早晨发了夜晚的雪景，但是由于是手机拍摄，效果渣得很，于是白天的时候又出去拍了几张照片。

毕竟是这个冬天里第一场像模像样的雪，并且雪后空气质量超好，支持出去转转也值得拍照留念。

封面这个红色的小消防车和纯白的雪形成了鲜明的对比，看起来有点意思吧？

夜、雪

昨天傍晚的时候突然下起了大雪，到了晚上八九点钟雪停了以后外边变得银装素裹分外美丽。

朋友圈/微信群里开启了晒雪景大赛，几乎我认识的且在本地的朋友都下楼拍照/堆雪人/晒雪景去了。

看着他们晒得热闹，在晚上十点半左右，我也忍不住从床上爬了起来，出去转了一圈，拍几张图片回来晒。

手机晚上拍照效果一般，不过总算赶上潮流了，不然看别人一直晒自己不晒，感觉有点对不起这场雪呢？

不过晒完雪景后，我 不禁陷入了深深的思索，为啥白天的时候欠儿登地去把车洗了呢？

Softlayer 彻底消失了

今天收到IBM标题为Event 95242588 - Permanent redirect of www.softlayer.com的邮件，内容如下：

On Wednesday, November 13th at 1:00pm CST, all URLs on www.softlayer.com will be redirected to their permanent replacements on ibm.com. No downtime is expected during this change.


(图源 ：pixabay)

大意是，从CST时间11月13日1:00pm开始，www.softlayer.com这个域名被永久定向到他们在ibm.com上的永久替代者。

刚刚测试了一下，访问softlayer.com已经被重定向到https://www.ibm.com/cloud， 这也意味着Softlayer这个品牌彻底的消失了。

尽管现在除了两台独服外的VPS，都放到Linode和Amazon，但是毕竟在SOFTLAYER用了近15年，感情还是挺深厚的，看它消失或者说蜕变，心里感觉还是挂怪的。

其实SOFTLAYER的前身是THEPLANET，大概04年的时候我是THEPLANET的用户，后来大概是06年前后，THEPLANET的一些资深工程师出去另立山头，成立了SOFTLAYER，我又去SOFTLAYER租了几台服务器。

然后戏剧的是SOFTLAYER又被IBM收购，被IBM收购后，SOFTLAYER又把THEPLANET收购，这样我的几台服务器又都跑一家去了。

SOFTLAYER被IBM收购后很长一段时间都保持独立运行（至少从网站都后台都是SOFTLAYER的系统），我一直很是推崇SOFTLAYER的系统，推荐了N个朋友购买。

到后来SOFTLAYER的后台换到IBM那里，我就觉得很难用了，当然可能更多是心里因素吧，我承认我比较怀旧。

直到收到今天这封邮件，我才直到SOFTLAYER这次是彻底消息了，了无痕迹。所以发篇文章，随便发些牢骚，为SOFTLAYER留下一点痕迹吧。

携号转网 & 移动的套路

虽然微信群以及朋友圈里很少讨论，但是最近新闻媒体炒作的一个热点就是携号转网，因为从11月10日起，移动、联通、电信三大运营商携号转网服务已经正式进入试运行阶段。


(图源 ：pixabay)

炒作了无数年的携号转网，终于可以操作了，但是其实身边的朋友们反应平平，因为该换号/换网的早就换了，不换的估计要么是懒惰，要么是对原来的运营商还是有些认同感的。

其实我也想不出对我而言携号转网有啥必要性，多个网络运营商终归多一些保障，经常有到哪个地方联通信号不好而移动信号好或者反过来的情况，如果只用一个运营商那么遇到有事情接打电话的时候就会很尴尬。

尽管我本人并不需要携号转网，但是对大众而言，多个选择，多些自由，终归是好事，不过话说携号转网真的很容易操作嘛？

好奇心的驱使下，两天前，我就做了一下测试，首先发送如下短信到10086

CXXZ#用户名#证件号码

当然了，要把用户名和证件号码换成自己的信息，其中CXXZ，据我分析应该是查询携转的缩写。

结果很快就收到10086的回复，果不其然，转网，想多了：

回复了Y之后，移动给出了原因：

擦，我的信息咋还不符了呢？还有网龄回馈流量扩容又是什么鬼？往前翻记录，大概想起来是怎么回事了， 大概8、9月份大家都在传发送1999给10086，就能获得免费流量，于是我也试了一下：

看来移动早早的就在给我们布套，然后我就傻傻的钻了进去。

当然了，估计联通、电信也可能有这样的政策/对策，不过为什么不从根源上解决问题呢，如果你网络稳定、服务好、价格便宜，那么还会担心用户携号跑路吗？

你要携号转网吗？你被套路了吗？

纸上得来终觉浅：lxd初体验

还记得我写过的lxc初体验嘛？在那篇帖子中，我曾经说过lxd是lxc的进化版，这篇文章中我们来测试一下lxd。


(图源 ：pixabay)

安装

如果还没有安装lxd，那么我们需要如下步骤来安装lxd并且让当前用户可以直接使用lxd的相关命令。

安装lxd

sudo apt install lxd

登出并重新登录当前用户，以刷新组成员信息，或者使用如下指令：

newgrp lxd

在一台新的VPS上，我进行这样的操作后，也无法直接只用诸如lxc list等命令，出现下列错误：


Error: Get http://unix.socket/1.0: dial unix /var/lib/lxd/unix.socket: connect: permission denied

只好手工把用户添加到组lxd中：

sudo usermod -a -G lxd $(whoami)

可以使用groups命令来检查关联组别是否成功。

lxd初始化

在正式开始之前，我们还需要执行lxd init来进行一些初始化设置（网络、存储等等）。

lxd init

我们都选择默认设置即可（空间我设置的5G，因为我磁盘空间比较紧张)：

查询/创建/使用容器

查询

在进行完上述设置后，我们可以使用如下命令查询我们机器上的lxd容器：

lxc list

可以看到我们现在一个容器都没有呢：

注：lxc并非之前文章中的lxc，本文中代表的是lxd的客户端。

创建

下面来创建我们的第一个lxd容器：

lxc launch ubuntu:18.04 first

创建后，容器会自动启动：

再来查询一下：

lxc list

耶，有了一个(我有些后悔，初始化的时候应该关闭ipv6，看着闹心）：

使用

我们可以使用如下命令进入到对应的容器中：

lxc exec first -- /bin/bash

或者执行命令，拉取文件，推送文件等：

lxc exec first -- apt-get update
lxc file pull first/etc/hosts .
lxc file push hosts first/tmp/

至于关闭和开启可以用如下命令：

lxc stop first
lxc start first

不需要了，可以用以下命令删除：

lxc delete first
