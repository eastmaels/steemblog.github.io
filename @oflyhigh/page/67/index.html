升级steemd程序至v0.22.1

昨晚临睡前1分钟，发现steemit的官方github上释出了steem的v0.22.1版本，吓得马上精神了。

不过去对比了一下修改，发现只不过是在原来修复BUG的基础上，修改了一下版本号。也就是说，只要我们运行的程序是20190901-fix-curation-weight加了这个补丁的，那么功能上和v0.22.1是没有什么区别的。

不过对于强迫症而言，这就不一样了，毕竟显示的版本号是有区别的嘛。所以当然要Builld一下啦！

编译完成，检查一下版本号啥的是个好习惯，万一编译了错误的版本呢？

programs/steemd/steemd --version

还好一切正常

尽管官方说从0.22.0到0.22.1需要重播(Replay)

Steem 0.22.1 does require reindexing from 0.22.0

如果你运行的原始的0.22.0，那么必须重播(Replay)。

但是对于我们这些打过补丁，并且已经重播(Replay)过的选手来讲，其实就是改个版本号，根本不需要重播的啦。

只需替换原有的steemd，并重启程序，就可以啦。

这样不需要replay的升级还是蛮舒服的，但愿这版稳定下来吧。

(图源 ：pixabay)

尽管很多见证人节点在几个小时内完成重播，让steem区块链重新运转起来，但是遗憾的是，steemit官方节点(api.steemit.com)很长时间未能修复，导致很多依赖于此节点的应用(包括steemit.com)无法正常运行。

悲催的是，我们的公众号程序，就是依赖于官方节点(api.steemit.com)，所以这两天公众号的信息一直不正常，要么是数据老旧，要么是显示：

Replay 测试

朋友的节点无意中触发了Replay，并且不显示Replay进度，让他极为闹心。


(图源 ：pexels.com)

我看STEEM源码中的一处注释，得出结论，Replay是可以暂停的，暂停后重启steemd（不加 --replay-blockchain）会自动从之前的位置继续Replay。

我的依据就是代码中的这段注释：

尤其是这句：

If the index file head is in the log, but not up to date, replay from index head.

结果朋友按我的馊主意测试后，程序又从头Replay了，害他浪费了几个小时的时间。

为了证实replay中途停掉再启动程序后是否会从停掉的位置继续Replay，我做了一些测试。

删除block_log.index 以及shared_memory.bin，只保留block_log，重启steemd：

以上操作会自动触发Replay，不过并不显示进度。

等一段时间后，关掉steemd，再重新启动，不加（–replay-blockchain）参数：

发现并没有如我所预料的那样，从停掉的位置开始。

重新试一遍上述两步操作，结果没有什么区别：

再测试一下加（–replay-blockchain）参数，然后停掉重启，结果出现下边第二幅图中显示的错误：

经过无数次的测试，后来有一次，貌似Replay从中途继续的，不过我没注意观察，随手关掉了，哭。

看来朋友的结论还是对的，只要需要replay，别报期望中途停止恢复啥的，直接加 --replay-blockchain参数，然后耐心等待好了。

破壁机还不错

之前有说用招行35290分的巨额积分换购了一个名为威泰戈 (WHITE TIGER) 的破壁机。

这些积分换购个破壁机是什么概念呢？就像相当于商场消费了70万5000千元，赠品送了个破壁机。( ╯□╰ )，所以算下来其实这玩意真的不便宜。

装上机器，用它自带的清洗功能清洗了几遍，发现这个功能倒是很省心。如果你用过豆浆机，就知道豆浆机的清洗，是多么令人抓狂的一件事情。

早晨用它打了一壶豆浆，发现打出来的豆浆很细腻，没有残渣。以前用九阳的豆浆机，要用滤网过滤到残渣才可以，麻烦不说，浪费了很多豆子 。

另外，破壁机还有其它搅拌、榨汁等诸多功能，留着以后慢慢发现吧。这玩意这么好玩，难怪朋友向我强烈推荐啦。
